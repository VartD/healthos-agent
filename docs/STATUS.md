# Реальный статус функций HealthOS

Все статусы основаны на проверке исходного кода и локальном ручном smoke-check (с использованием SQLite). **Автоматизированные тесты (unit, integration, E2E) в проекте полностью отсутствуют.**

| Функция | Статус | Комментарий |
|---------|--------|-------------|
| **Регистрация и идентификация** | Работает частично | Нет явной регистрации. `user_id` берется из Telegram `chat.id` и передается в API. Проверено локально. |
| **Утренний брифинг / вечерний чекин** | Отсутствует | Нет планировщика (APScheduler) и логики проактивных сообщений. |
| **Ввод воды, глюкозы, мочевой кислоты, сауны** | Работает (без валидации) | Реализовано через команды `/water`, `/glucose`, `/uric`, `/sauna` с парсингом числа. Отсутствует валидация отрицательных и физиологически невозможных значений. Проверено локально. |
| **Питание, тренировки, симптомы** | Работает | Реализовано через команды `/food`, `/workout`, `/symptom` с парсингом текста (note). |
| **Сон, давление, вес, лекарства** | Отсутствует в боте | Модель БД поддерживает эти типы событий, но в боте нет команд для их ввода. |
| **Загрузка фотографий / Vision-анализ** | Отсутствует | Бот не обрабатывает изображения. |

| **Problem List / Next Best Action** | Работает частично | Вместо сложных AI-агентов работает детерминированный `risk_engine` и `command_engine`, выдающий простые команды (например, "стоп кофе", "вода"). Проверено локально. |
| **Weekly GP review / MDT-консилиумы** | Отсутствует | Нет AI-агентов и логики ревью. |
| **PubMed-цитаты** | Отсутствует | |
| **Экспорт CSV/JSON** | Отсутствует | Нет эндпоинтов или команд бота. |
| **Уведомления / Восстановление scheduler jobs** | Отсутствует | Планировщика нет. |
| **Mini App / React Native приложение** | Отсутствует | Кода нет в репозитории. |
| **Digital Twin / Longevity Intelligence** | Отсутствует | |

## AI-Агенты

**Действительно ли работают все 14 агентов?**
Нет. Ни один AI-агент не реализован. В коде нет интеграции с OpenAI, Anthropic или другими LLM-провайдерами.
Вместо агентов используется **rule-based система** (жестко закодированные if/else правила в Python):
- Вычисление режима (STABILIZATION, TRAINING и т.д.) по порогам.
- Вычисление рисков (например, глюкоза > 7).
- Сборка текстовых строк из словарей.

**Оркестратор, трассировки, тесты AI**
Отсутствуют.

**Red flags и safety layer**
Присутствует базовая детерминированная проверка в `analysis.py`: функция `_severe_symptom` ищет ключевые слова ("боль в груди", "обморок", "инфаркт" и т.д.) в заметках к событиям. Если находит — добавляет дисклеймер с рекомендацией обратиться к врачу. Ограничения медицинских рекомендаций также прописаны статичным текстом в `/start` и в конце каждого дайджеста.
