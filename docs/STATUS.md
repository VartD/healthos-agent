# Реальный статус функций HealthOS

Статусы основаны на проверке кода, smoke-check и автоматизированных тестах с SQLite.
После Telegram integration-этапа выполняются 23 теста. Реальный Telegram polling
и прикладной E2E подтверждены 10 июля 2026 года. Docker runtime и backup/restore
пока не подтверждены логами.

## P0 foundation, добавленный Codex

- все health-data endpoints требуют `X-API-Key`;
- `GET /events` требует конкретный `user_id`, массовая выдача отключена;
- отрицательные и заведомо невозможные числовые значения отклоняются;
- бот передаёт API-key в backend;
- исправлено использование максимального значения вместо последнего измерения;
- единицы воды больше не теряются перед risk engine;
- критический symptom disclaimer ограничен последними 24 часами;
- часовой пояс вынесен в `HEALTHOS_TIMEZONE`;
- добавлен структурированный модуль сна, профиль и 3 Telegram-команды;
- добавлены 23 автоматизированных теста;
- добавлены Alembic, CI, health checks, JSON request logs и backup/restore scripts;
- Docker Compose расширен до PostgreSQL + backend + bot.

## AI-Агенты

**Действительно ли работают все 14 агентов?**
Нет. Ни один AI-агент не реализован. В коде нет интеграции с OpenAI, Anthropic или другими LLM-провайдерами.
Вместо агентов используется **rule-based система** (жестко закодированные if/else правила в Python):
- Вычисление режима (STABILIZATION, TRAINING и т.д.) по порогам.
- Вычисление рисков (например, глюкоза > 7).
- Сборка текстовых строк из словарей.

**Оркестратор, трассировки, тесты AI**
Отсутствуют.

**Red flags и safety layer**
Присутствует базовая детерминированная проверка в `analysis.py`: функция `_severe_symptom` ищет ключевые слова ("боль в груди", "обморок", "инфаркт" и т.д.) в заметках к событиям. Если находит — добавляет дисклеймер с рекомендацией обратиться к врачу. Ограничения медицинских рекомендаций также прописаны статичным текстом в `/start` и в конце каждого дайджеста.
